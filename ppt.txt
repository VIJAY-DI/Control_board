Experiment I:
Local Version Control using Git
Objective: Demonstrate local version control configurations and workflows using Git.

Tools Required: Git, Python, Visual Studio Code (VSCode)

Step 1: Create a Project Directory
Explanation: Before using Git, you need an ordinary folder on your computer to store your files.

What to do: Open VSCode, click File > Open Folder, right-click in your file window, create a new folder named Exp1, and click Select Folder.

Step 2: Open the Terminal
Explanation: Git runs by typing text commands into a command-line interface. VSCode provides a built-in terminal directly inside your project window.

What to do: Go to the top menu of VSCode and click Terminal > New Terminal.

Step 3: Initialize Git and Check Status
Explanation: An ordinary folder cannot track code updates. Running git init creates a hidden database that tells Git to start watching every line of code inside this folder.

What to type into the Terminal:

Bash
git init
git status
Step 4: Write the Application Code
Explanation: Create a simple Python script to use as your first project file.

Where to write it: Create a new file in VSCode named greet.py and write this code:

Python
# Basic Python Program - Version 1
print("Hello, World!")
What to do: Save the file by pressing Ctrl + S.

Step 5: Stage and Commit the Code
Explanation: Git uses a two-step saving process. git add puts your file into a temporary "staging sandbox." git commit takes a permanent historical snapshot of that staged data.

What to type into the Terminal:

Bash
git add greet.py
git commit -m "Initial commit: add greet.py"
git status
Step 6: Modify the File to Create Version 2
Explanation: To see how Git tracks updates, modify your file to create a second milestone.

Where to write it: Open your greet.py file again, change the text to look like this, and press Ctrl + S:

Python
# Basic Python Program - Version 2
print("Hello, Git versioning!")
Step 7: Save the Second Version in Git
Explanation: Since you changed the file, you must stage and commit it again to register the new version in Git's history log.

What to type into the Terminal:

Bash
git add greet.py
git commit -m "Updated greet.py version2"
Step 8: View the History Log
Explanation: Running git log shows a timeline of all your saved milestones, including who made them, when they were made, and their unique alphanumeric Commit IDs.

What to type into the Terminal:

Bash
git log
Step 9: Revert to a Previous Version
Explanation: If your code breaks, Git allows you to roll your files back to any previous milestone. Look at your terminal screen from Step 8, find the long ID code for your "Initial commit", and use it in the checkout command.

What to type into the Terminal:

Bash
git checkout <insert_your_initial_commit_id_here> greet.py
(Example: git checkout belecb4be35c76f92a18887dcab39f3bbea63afb greet.py)

Experiment II:
Cloud Repositories, Branching, and Merging on GitHub
Objective: Configure remote repositories on GitHub, handle SSH keys, manage independent branches, and resolve merges.

Tools Required: Git, a free GitHub Account, Python, VSCode

Step 1: Create a Cloud Repository
Explanation: GitHub hosts your code repositories online so your work is backed up in the cloud.

What to do: Open your browser, log in to github.com, click the + icon at the top right, choose New repository, name it Exp2, keep it Public, and click Create repository.

Step 2: Set Up an Independent Local Workspace
What to do: In VSCode, open a new folder named Exp2 and launch a fresh terminal screen (Terminal > New Terminal).

What to type into the Terminal:

Bash
git init
Step 3: Write your First Script
Where to write it: Create a file named num.py in VSCode and write this script:

Python
# Print numbers (without input)

Step 4: Link Your Machine to GitHub Using SSH Keys
Explanation: GitHub requires authorization before allowing you to upload files. Generating an SSH key creates a secure, password-free bridge between your local computer and your online account.

What to type into the Terminal:

Bash
ssh-keygen -o
(Press Enter three times to skip the default passphrases)

How to display your key: Type the following command to print your public cryptographic key on screen:

Bash
cat ~/.ssh/id_ed25519.pub
What to do: Highlight and copy the entire text line that appears on your terminal screen. Go to your GitHub browser tab, click your profile photo at the top right, and click Settings > SSH and GPG keys > New SSH Key. Name it Clientkey1, paste the copied code into the Key box, and click Add SSH key.

Explanation: Upload your local repository data up into your blank remote GitHub profile.

What to type into the Terminal:

Bash

git remote add origin https://github.com/your_username/Exp2.git
git push -u origin main
(Replace your_username with your real GitHub account name)

Step 6: Create an Isolated Feature Branch
Explanation: Working on the production timeline is risky. Creating a development branch (named dev) lets you test features safely without breaking your master codebase.

What to type into the Terminal:

Bash
git switch -c dev
git branch
Step 7: Write a New Feature File
Explanation: This file will only exist inside your isolated development environment for now.

Where to write it: Create a file named num1.py in VSCode and write this interactive script:

Python
# Print first n numbers (with input)

for i in range(1, n + 1):
    print(i)
What to do: Save the file (Ctrl + S), and commit it to your branch history by running this in the terminal:

Bash
git add num1.py
git commit -m "num1.py code with inputs"
Step 8: Return to Main and Merge the Features
Explanation: When you switch back to the main branch, num1.py will temporarily disappear from your sidebar view. Running git merge dev seamlessly incorporates your newly completed feature file into the primary production line.

What to type into the Terminal:

Bash
git switch main
git merge dev
git push -u origin main

Experiment III: Containerization with Docker (Flask Microservice)
Objective: Understand container runtimes, construct build recipes, and launch an isolated Flask application.

Tools Required: Docker Desktop engine, Python, VSCode

Step 1: Open Your Environment Workspace
What to do: Create a new folder named Exp3 on your machine, open it in VSCode, and open a new terminal window.

Step 2: Define Library Dependencies
Explanation: To make sure anyone running this application automatically installs the correct libraries, specify them in a standard requirements document.

Where to write it: Create a text file named requirements.txt and write this line:

Plaintext
flask==2.3.2
Step 3: Write the Web Application Server Script
Explanation: This lightweight web app listens for connections and displays a greeting whenever a user visits it.

Where to write it: Create a file named app.py and write this script:

Python
from flask import Flask

app = Flask(__name__)

@app.route("/")
def home():
    return "Hello, Dockerized Flask App!"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
Step 4: Write the Dockerfile Build Recipe
Explanation: A Dockerfile is a blueprint containing step-by-step instructions for Docker to compile a completely self-contained environment for your application.

Where to write it: Create a text file named exactly Dockerfile (Capital 'D', and no extension like .txt) and write these lines:

Dockerfile
# Use an official lightweight Python operating system image
FROM python:3.9-slim

# Set the active working directory inside the container
WORKDIR /app

# Copy application configuration files into the container
COPY app.py .
COPY requirements.txt .

# Run the python installer to download dependencies inside the container
RUN pip install -r requirements.txt

# Open the network port inside the container
EXPOSE 5000

# Specify the command to start the app when the container opens
CMD ["python", "app.py"]
Step 5: Build the Docker Image
Explanation: Compile your application and its dependencies into a single, reusable package (an image). The dot . means your Dockerfile is in your current directory.

What to type into the Terminal:

Bash
docker build -t flask-web-app .
Step 6: Launch Your Application Container
Explanation: Run the compiled image as an isolated container in the background (-d). Mapping port 5000:5000 tells your local computer to send any web traffic to port 5000 directly into the container.

What to type into the Terminal:

Bash
docker run -d -p 5000:5000 --name flask-container flask-web-app
Step 7: Test Your Application
What to do: Open any web browser and go to this address: http://localhost:5000. You should see the message: "Hello, Dockerized Flask App!" clear on your screen.

Step 8: Clean Up Resources
What to type into the Terminal:

Bash
docker ps
docker stop flask-container
docker rm flask-container

Experiment IV: Multi-Container Architectures with Docker Compose
Objective: Link multiple services together, map environment configurations, and handle persistent database data storage volumes.

Tools Required: Docker Desktop, Docker Compose Engine, Python, VSCode

Step 1: Open Your Workspace Directory
What to do: Create a folder named Exp4, open it inside VSCode, and launch a new terminal panel.

Step 2: Write the Relational Database Application Script
Explanation: This updated script talks to an independent MySQL database container, sets up a table structure, saves user milestones, and displays the results on your browser.

Where to write it: Create a file named app.py and write this script:

Python
import mysql.connector
import time
from flask import Flask

app = Flask(__name__)

def get_db_connection():
    while True:
        try:
            conn = mysql.connector.connect(
                host="db",
                user="root",
                password="root123",
                database="testdb"
            )
            return conn
        except:
            print("Waiting for database container to initialize...")
            time.sleep(2)

@app.route('/')
def home():
    conn = get_db_connection()
    cursor = conn.cursor()
    cursor.execute("CREATE TABLE IF NOT EXISTS users (id INT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(255))")
    cursor.execute("INSERT INTO users (name) VALUES ('Docker User')")
    conn.commit()
    cursor.execute("SELECT * FROM users")
    rows = cursor.fetchall()
    conn.close()
    return f"Database Connected Successfully! Data: {rows}"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
Step 2: Define Library Requirements
Where to write it: Create a file named requirements.txt and list these two Python modules:

Plaintext
flask
mysql-connector-python
Step 3: Write the Core Dockerfile Blueprint
Where to write it: Create a file named exactly Dockerfile and add these build instructions:

Dockerfile
FROM python:3.9
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]
Step 4: Write the Docker Compose Service Orchestration Blueprint
Explanation: Manually configuring multiple containers is tedious. A docker-compose.yml file acts as a coordinator, launching both your application container and your database container at the same time. It also sets up a persistent storage volume (db_data) so your data isn't erased when the database stops.

Where to write it: Create a file named exactly docker-compose.yml and write this configuration layout:

YAML
version: '3'

services:
  web:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - db

  db:
    image: mysql:5.7
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: testdb
    ports:
      - "3307:3306"
    volumes:
      - db_data:/var/lib/mysql

volumes:
  db_data:
Step 5: Launch the Entire Architecture
Explanation: This spins up both your application container and your database container simultaneously. The --build flag ensures your source code updates are compiled before the containers launch.

What to type into the Terminal:

Bash
docker-compose up --build
How to verify: Open your browser and navigate to http://localhost:5000. Every time you refresh the page, you'll see new data logs being added to the database. Press Ctrl + C in your terminal window when you're done to shut down the environment.

Experiment V: Automated CI/CD Testing via GitHub Actions
Objective: Configure automated CI validation checks, integrate Pytest unit tests, and execute cloud container builds.

Tools Required: Git, Python, GitHub Repository, VSCode

Step 1: Open Your Workspace Folder
What to do: Create a folder named Exp5, open it in VSCode, and open a terminal panel.

Step 2: Write the Core Application
Explanation: Write a function that checks if words are palindromes (words that read the same backwards and forwards), which will serve as the codebase for this experiment.

Where to write it: Create a file named app.py and write this script:

Python
def is_palindrome(word: str) -> bool:
    cleaned = word.lower().replace(" ", "")
    return cleaned == cleaned[::-1]

if __name__ == "__main__":
    user_input = input("Enter a word: ")
    if is_palindrome(user_input):
        print(f"'{user_input}' is a palindrome!")
    else:
        print(f"'{user_input}' is not a palindrome.")
Step 3: Write Automated Local Unit Tests

    assert app.is_palindrome("racecar") is True

def test_palindrome_false():
    assert app.is_palindrome("hello") is False
    assert app.is_palindrome("python") is False

Step 6: Create Your GitHub Actions Automation Path
Explanation: GitHub Actions looks for automation instructions inside a very specific folder path. You'll create a pipeline workflow file (ci.yml) that triggers automatically every time you push code, running your test suite and verifying your container build on a cloud server.

What to type into the Terminal to build the folders:

Bash
mkdir -p .github/workflows
Where to write it: Create a new configuration file inside that folder path named .github/workflows/ci.yml and write this script:

YAML
name: CI/CD Pipeline

on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: "3.10"

      - name: Install dependencies
        run: |
          pip install -r requirements.txt

      - name: Run tests
        run: python -m pytest

  docker-build:
    runs-on: ubuntu-latest
    needs: build-test
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Build Docker image
        run: docker build -t palindrome-app .

Step 5: Write the Advanced Multi-Job Automation Workflow
Explanation: This pipeline splits your automation into separate tasks: first it checks for style formatting using Flake8, then it enforces code coverage rules (failing the pipeline if test coverage falls below 80%), and finally it initializes an automated deployment job upon total success.

What to type into the Terminal:

Bash
mkdir -p .github/workflows
Where to write it: Create a file named .github/workflows/ci-cd.yml and write this pipeline configuration:

YAML
name: Python CI/CD Pipeline

on:
  push:
    branches: ["main"]
  pull_request:
    branches: ["main"]

jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.10"

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: Lint code (Quality Check)
        run: |
          flake8 . --count --select=E9,F63,F7,F82 --show-source --statistics
          flake8 . --count --exit-zero --max-complexity=10 --max-line-length=127 --statistics

      - name: Run tests with coverage
        run: |
          pytest --cov=app --cov-report term --cov-fail-under=80

      - name: Validate script output
        run: |
          output=$(python app.py)
          if [ "$output" != "5" ]; then
            echo "Unexpected output: $output"
            exit 1
          fi

  deploy:
    needs: build-test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy Application
        run: echo "Deploying application..."

      - name: Post-deployment validation
        run: echo "Deployment successful and verified!"
